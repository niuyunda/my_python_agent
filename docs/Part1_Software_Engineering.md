# 第一部分：什么是软件工程？它和 IT 项目管理的不同是什么

你好！欢迎来到软件工程的世界。很多人刚开始编程时，都会产生类似的疑问：“我已经能写出可以跑通的 App 和网站了，这难道不就是软件工程吗？”

其实，**写代码（Programming）** 和 **软件工程（Software Engineering）** 有着本质的区别。

## 1. 软件工程与“写代码”的区别

| 维度 | 写代码 (Programming) | 软件工程 (Software Engineering) |
| --- | --- | --- |
| **关注点** | 解决当前的问题，让代码立刻运转起来。 | 关注代码在**几周、几个月、甚至几年后**是否能正常运转，是否能随着需求变化而扩展。 |
| **参与人数** | 通常是 1 人，至多 2-3 人。 | 数十人、几百人甚至上千人同时协作，涉及不同的角色。 |
| **生命周期** | 一般短平快，写完就不用了，或者很少大规模修改。 | 代码的生命周期很长，需要不断地维护、修改、重构。 |
| **度量标准** | 能不能跑通？有没有 Bug？ | 可读性、可测试性、可维护性、可扩展性、安全性。 |

*一句话总结：写代码是工程师把想法告诉计算机；而软件工程，是把想法告诉**未来的自己和其他工程师**，同时让计算机执行。*

## 2. 为什么个人项目通常不算真正的软件工程？

在个人项目中，你通常包揽了产品经理、设计师、开发、测试和运维的所有角色。
在个人项目中，你的代码都在自己的大脑里，你清楚地知道每个变量是什么意思，这导致：

1. **缺乏规范和契约**：不需要 PR (Pull Request)，因为你直接推送到 `main` 分支。没有接口文档，因为前后端都是你写的。
2. **缺乏生命周期管理**：很少写单元测试，因为大不了自己手动点一点就能测出 Bug。
3. **脆弱的演进能力**：当你过半年再看自己的代码，或者要重构底层逻辑时，因为没有测试保护，改动非常容易引发连锁崩溃。这就是技术债。

**真正的软件工程，是设计一套机制（流程、规范、架构、自动化），使得一批普通水平的工程师协作，能够长期、稳定地交付高质量的复杂系统。**

## 3. 软件工程的核心思想

在软件工程中，有几个核心思想是你必须掌握的：

- **模块化与解耦 (Modularity & Decoupling)**：不要把所有代码写在一个文件里。系统应该分层（如后面要讲的 Controller/Service/Repository 层），每一层各司其职。
- **可测试性 (Testability)**：代码必须是能被自动化测试的。如果你发现一段网络请求或者数据库代码很难写测试，那说明它的耦合度过高了。
- **自动化 (Automation)**：人总是会犯错的。不要依赖人去手工编译、打包、测试。所有的编译、代码规范检查 (Lint)、测试、部署，都必须交给 CI/CD (Continuous Integration / Continuous Deployment) 管道自动完成。
- **版本控制与可追溯 (Version Control & Traceability)**：每一次代码的改动，都必须对应一个具体的需求或 Bug Ticket；每一行代码的变更，都能通过 Git 历史追溯到为什么改、谁改的、改了什么。

## 4. 软件工程与 IT 项目管理的区别

- **IT 项目管理 (Project Management)**：关注**成本、时间、范围**（所谓的“项目管理铁三角”）。项目经理 (PM / Scrum Master) 关心的是：“这功能什么时候能上？”、“现在开发进度阻塞在哪了？”、“今天谁请假了？”。
- **软件工程 (Software Engineering)**：关注**质量、架构、工程效率**。技术负责人 (Tech Lead) 关心的是：“这个功能的架构以后能抗住一万并发吗？”、“代码覆盖率够不够？”、“如何降低技术债？”。
